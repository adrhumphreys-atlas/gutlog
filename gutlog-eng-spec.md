## Problem Statement

People dealing with food-related gut issues (IBS, food intolerances, FODMAP sensitivities, etc.) need to track what they eat alongside their symptoms, bowel movements, emotions, and daily impact to identify trigger foods and patterns. Existing solutions are either calorie-counting fitness apps that ignore symptoms, or clinical gut health apps that feel like medical forms. GutLog is a personal food & symptom diary with a statistical correlation engine that identifies trigger foods automatically.

## Tech Stack

- **Frontend:** React 18 SPA (Vite-built), TypeScript, Tailwind CSS
- **Backend:** Hono (Cloudflare Workers), REST API routes
- **Database:** Cloudflare D1 (SQLite at the edge) via Drizzle ORM
- **Auth:** Custom magic link auth — D1-backed tokens, Resend for email, signed JWT session cookies. No passwords.
- **Validation:** Zod (discriminated unions for polymorphic entry types)
- **Charts:** Recharts or Chart.js for insights visualisation
- **Hosting:** Cloudflare Workers (API) + Cloudflare Pages (React SPA) + D1 (database)

## Data Model

```
User
├── id, email, createdAt, settings (JSON — timezone, preferences only)

├── id, userId, timestamp, type (enum: meal|symptom|bowel|emotion|impact|note)

├── Meal fields (nullable, used when type = 'meal'):

│   severity (1-5), location (upper_left/upper_right/lower_left/lower_right/central/all)

│
├── Bowel fields (nullable, used when type = 'bowel'):
│   bristolType (1-7), urgency (none/mild/moderate/severe)

│
├── Emotion fields (nullable, used when type = 'emotion'):
│   mood (1-5), stressLevel (1-5)

│
└── Impact fields (nullable, used when type = 'impact'):
    impactSeverity (none/mild/moderate/severe)
    affectedActivities (JSON array)
    description (free text)

Correlation (statistical insights — computed on-demand)
├── id, userId, triggerType (food/emotion/time_of_day/combination)
├── triggerValue, symptomType, confidence (0-1)
├── relativeRisk, consistencyRatio
├── occurrences, totalOpportunities, windowHours
├── createdAt, lastUpdatedAt

├── id, userId, name, description, status (active/completed/abandoned)
├── eliminatedFoods (JSON array — multi-term list, e.g. ["dairy", "milk", "cheese", "yogurt"])
├── startDate, endDate, durationDays
├── baselineSymptomRate, currentSymptomRate
├── result (significant_improvement/mild_improvement/no_change/worsened/null)
```

### Design Decisions (from eng review)

- **STI over separate tables:** Simpler queries for timeline view (one SELECT, no JOINs). Type-specific required fields enforced at application layer (Zod + Drizzle custom validators).
- **No FoodItem table:** Food autocomplete and frequency data derived directly from MealEntry.foods[] using queries. Single source of truth, no sync bugs.
- **No smart defaults cache:** Smart defaults (recent foods, last symptom location, preferred portion) derived at read time by querying recent entries. No User.settings cache, no race conditions.
- **JSON as TEXT:** D1/SQLite stores JSON columns as TEXT. Drizzle serializes/deserializes transparently. Functionally identical for `foods[]`, `affectedActivities[]`, `eliminatedFoods[]`, `settings`.
- **TEXT enums:** SQLite has no native enum type. Entry `type`, `mealType`, `symptomType`, etc. stored as TEXT columns. Zod enforces valid values at the application layer (which it was already doing).

## Correlation Engine

### On-Demand Smart Trigger

No cron jobs. Correlations refresh automatically when the user opens the Insights page and new data exists since the last refresh.

```
User opens /insights
  ├── Is lastEntryAt > lastCorrelationRefresh?
  │   YES → run correlation refresh (below), update lastCorrelationRefresh
  │   NO  → return cached Correlation records
  └── Manual "Refresh" button also available (rate-limited 3/day)
```

### Simplified Scoring (Relative Risk)

For each food item logged ≥3 times:

```
relative_risk = P(symptom within N hours | ate food X)
              / P(symptom within N hours | did NOT eat food X)

consistency = occurrences / totalOpportunities
              (e.g., 4 symptoms out of 5 times eating bread = 0.8)

sample_size_factor = min(occurrences / 10, 1.0)

confidence = (relative_risk_normalized * 0.4)
           + (sample_size_factor * 0.3)
           + (consistency * 0.3)

GUARD CLAUSE: If P(symptom|¬food) = 0 (division by zero),
              use consistency ratio as confidence instead.
```

- Correlations below 0.3 confidence are not shown
- Minimum 3 occurrences required
- Cross-reference with emotion data: stress amplification analysis
- Time-of-day analysis: symptom patterns by meal timing

### No LLM / No AI

V1 shows statistical correlations directly — bar charts, percentages, trend lines. No natural language summaries. No AI experiment suggestions. Users interpret the data themselves.

## Experiment System

### Multi-Term Elimination Lists

When creating an experiment, users specify multiple food terms to watch for:

```
Example: Eliminate "dairy"
→ eliminatedFoods: ["dairy", "milk", "cheese", "yogurt", "butter", "cream"]
```

Compliance check scans all meal entry foods[] against the full elimination list (case-insensitive exact match on each term).

### Experiment Lifecycle

```
Create experiment
  → Set eliminatedFoods[] (multi-term), duration
  → Calculate baseline symptom rate (last N days before start)

During experiment
  → Detect compliance violations (eliminated food in meal log)
  → Track current symptom rate
  → Show progress (day X of N)

End experiment
  → Compare baseline vs elimination symptom rates
  → Classify result:
     significant_improvement (>50% drop)
     mild_improvement (20-50% drop)
     no_change (<20% change)
     worsened (increase)
  → Mark status = completed

Abandon experiment
  → Mark status = abandoned
```

## Auth — Custom Magic Link (Cloudflare Workers)

- Custom implementation (~100 LOC) — no NextAuth dependency (requires Node.js runtime, incompatible with CF Workers)
- **AuthToken table in D1:** `id`, `email`, `token` (crypto-random), `expiresAt` (15-min TTL), `usedAt` (null until consumed)
- **Flow:** `POST /api/auth/send` → generate token, store in D1, send email via Resend → user clicks link → `GET /api/auth/verify?token=` → validate token (exists, not expired, not used) → mark as used → create/find User → set signed JWT session cookie (HttpOnly, Secure, SameSite=Lax)
- **Session validation:** Hono middleware checks JWT on every authenticated request. JWT contains `userId` and `exp`. No DB lookup needed for session validation.
- **Token security:** Single-use (marked `usedAt` on verification), 15-min TTL, crypto-random (128-bit)
- Requires email service: Resend (free tier — 100 emails/day, more than enough)
- No passwords, no password reset flow
- Multi-user ready (User created on first magic link verification if email not found)
- **Email failure error handling:** Catch Resend API errors, surface "Failed to send — try again" to user instead of showing "Check your email" on failure

## Input Validation (Zod)

- All entries require: `userId`, `timestamp`, `type`. Reject with 400 if missing.
- `timestamp`: valid ISO 8601, not in the future (5 min grace for clock skew)
- Severity/mood/stress scales: integer 1-5 only
- Bristol stool type: integer 1-7 only
- `mealType`: one of `breakfast`, `lunch`, `dinner`, `supper`, `snack`
- `foods[]`: max 20 items per meal, each item max 200 characters
- Free-text fields (`notes`, `description`): max 1000 characters
- Use Zod discriminated union on `type` field to validate type-specific required fields

## API Error Responses

- `400` — Validation error (with field-level error messages)
- `401` — Not authenticated
- `403` — Accessing another user's data
- `404` — Entry/experiment not found
- `429` — Rate limited (60 req/min general, 3/day for insight refresh)
- `500` — Internal server error (logged, generic message to client)

## Timezone Handling

All timestamps stored in UTC. User's timezone in `User.settings.timezone` (IANA format). Frontend sends browser timezone on first login, manual override in Settings. All display converts UTC → user timezone. Correlation engine uses user timezone for time-of-day buckets and day boundaries.

## Food Autocomplete

- Activates after ≥5 meal entries logged
- Suggestions derived from MealEntry.foods[] (query D1, don't cache):
  - Ranked by: (1) frequency, (2) recency, (3) `LIKE` prefix/substring match (`COLLATE NOCASE`)
  - Max 5 suggestions
- Recent chips: top 3 most-logged foods from last 7 days, tappable for instant entry

## Data Export & Import

- **Export:** JSON and CSV via `/api/export` (all data or filtered by date/type)
- **Import:** `/api/import` accepts same JSON format as export
  - Full validation on each entry before inserting
  - Duplicate detection by `timestamp + type + userId`
  - TODO: Add batch processing (chunks of 100) for large imports. D1/Workers free plan has ~1MB request body limit — chunking is required for large datasets.

## UI/UX Design Principles

1. **Emoji-first inputs** — Tap emoji to set severity, mood, meal type
2. **Smart defaults** — Pre-fill time to "now", suggest recent foods, remember last symptom location (all derived at read time)
3. **Timeline as home** — Day at a glance, all entry types color-coded
4. **Progressive disclosure** — Required fields first, optional details expandable
5. **Warm, not clinical** — Soft greens/earth tones, rounded corners, personal language
6. **Celebration of streaks** — 🌿 streak counter on timeline header, tap for calendar heatmap
7. **Bottom sheets, not pages** — All entry forms open as modals over timeline. Logging is annotation, not context switch.
8. **Honest labeling** — "Pattern Spotter" not "AI Detective." Statistical patterns, not AI.
9. **WCAG AA accessibility** — Contrast ≥4.5:1, 44px touch targets, ARIA labels, focus traps, keyboard nav
